# ハーネスドキュメント生成・更新

実装完了後に、プロジェクト全体で一貫した粒度・配置でハーネスドキュメントを生成・更新するワークフロー。

## いつ使うか

- プランの実装が完了した後
- 新機能を追加・大幅変更した後
- アーキテクチャや設計方針を変更した後
- 「ドキュメント化して」「ハーネス更新して」と言われた時

## 基本原則

### ドキュメントの3層モデル

| 層 | 役割 | 配置先 | 例 |
|---|------|--------|-----|
| **指示** | 「こうしろ / するな」 | `.claude/rules/` | コーディング規約、データ整合性ルール |
| **インデックス** | 「何が重要か / どこを読め」 | `CLAUDE.md` | パッケージ概要、重要な前提、`@` 参照 |
| **知識** | 「なぜこうなっているか」 | `docs/` / feature `README.md` | 設計背景、データフロー、依存関係、状態遷移 |

### 書くべきもの / 書くべきでないもの

**書く:**
- コードから読み取れない設計判断とその理由
- 暗黙の依存関係（バッチ処理、外部サービス、ワーカー等）
- 複数レイヤーにまたがるデータフロー
- 非自明な制約や前提条件
- エッジケースや落とし穴

**書かない:**
- コードから直接読み取れること（関数のシグネチャ、型定義の列挙等）
- git history から得られること（誰がいつ変更したか）
- 標準的な言語・フレームワークの慣例
- 自明な事実（「このファイルは X を export している」等）

## ワークフロー

### ステップ1: 変更のスコープを把握する

git diff / git log で実装した変更の全体像を収集し、以下を判断する:
- どのパッケージ / ディレクトリが影響を受けたか
- 新しいパッケージやディレクトリが追加されたか
- アーキテクチャレベルの変更か、機能レベルの変更か

併せて、セッション中にツールやスキルが生成した**作業成果物**（プランファイル、作業ログ、AI生成ドキュメント等）がないか探す。これらはコミット可能な構造化ドキュメントではないが、実装意図・設計判断・タスク分解の経緯など**ドキュメント化の貴重なソース**になる。

典型的な作業成果物の例:
- `docs/<skill-name>/` 配下のスキル生成ファイル（superpowers 等）
- `.claude/` 配下のプランファイル
- セッション中に作成された一時的な設計メモ

これらは**参照ソースとして読むが、そのまま最終ドキュメントにはしない**。内容を精査し、3層モデルに従って適切な場所に構造化して書き直す。

### ステップ2: 既存ドキュメントの現状を調査する

変更に関連する既存ドキュメント（CLAUDE.md, .claude/rules/, docs/, feature README）をすべて読む。

確認ポイント:
- 既存ドキュメントの粒度・トーン・構成を把握する
- 今回の変更で古くなったドキュメントがないか特定する
- 重複しそうな記述がないか確認する

### ステップ3: 配置判断

変更内容を分類し、何をどこに書くか決定する。
具体的な判断フローと配置先マッピングは [references/placement-guide.md](references/placement-guide.md) を参照。

### ステップ4: 一貫性チェック

ドキュメントを書く前に、既存ドキュメントとの一貫性を確認する。

- **粒度**: 既存の同レベルドキュメントと同じ詳細度か
- **構成**: 既存のセクション構成・見出しレベルに揃えているか
- **トーン**: 既存の文体（体言止め / 文末表現 / 言語）に合わせているか
- **参照**: `CLAUDE.md` の `@` 参照を更新したか
- **重複**: 他のドキュメントと内容が被っていないか

### ステップ5: ドキュメントの生成・更新

以下の優先順位で作業する:

1. **既存ドキュメントの更新**（古くなった記述の修正、新情報の追記）
2. **CLAUDE.md の `@` 参照追加**（新しい docs/ ファイルへのリンク）
3. **新規ドキュメントの作成**（必要な場合のみ）

#### 配置先ごとのルール

**CLAUDE.md:**
- **200行以下**に収める。詳細は `@` 参照で指し示す
- 「何が重要か」「どこを読めばいいか」だけを伝える

**.claude/rules/:**
- 1ファイル = 1トピック。`paths` frontmatter で適用範囲を限定する
- 「こうしろ / するな」の行動指示に徹し、理由（why）を簡潔に添える
- 知識や設計背景は書かない（それは docs/ の役割）

**docs/:**
- 人間も読むドキュメントとして書く（Claude 専用にしない）
- 「なぜこうなっているか」の設計背景を中心に
- ファイル名は内容を表す具体的な名前にする（同階層に複数ある場合）

**feature README.md:**
- その機能の全体像を1ファイルで把握できるようにする
- レイヤー間の依存関係、非自明な設計判断とその理由、暗黙の前提（バッチ処理、外部ワーカー等）を必ず記載する

### ステップ6: 最終確認

すべてのドキュメント更新が完了したら、以下を確認する:

- [ ] CLAUDE.md が 200 行以下に収まっているか
- [ ] 新しい docs/ ファイルが CLAUDE.md から `@` 参照されているか
- [ ] `.claude/rules/` の `paths` が正しいパターンか
- [ ] 既存ドキュメントとの重複がないか
- [ ] コードから読み取れることを冗長に書いていないか
- [ ] 変更で古くなった既存ドキュメントを更新したか

確認結果をユーザーに報告し、生成・更新したファイルの一覧と各ファイルの概要を提示する。

## monorepo での注意点

- パッケージ固有のコンテキスト → パッケージ直下の `CLAUDE.md`（ルートの `.claude/rules/` ではない）
- 横断的な規約のみルートの `.claude/rules/` に配置する
- パッケージの `CLAUDE.md` もインデックスに徹し、詳細は `docs/` に分離する
